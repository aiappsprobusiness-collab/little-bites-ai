# Recipe Pool — Trust Workflow (Stage 2.5)

Операционный документ: управление качеством пула рецептов через feedback и trust level.

---

## Принцип

- **candidate** — черновик (новые AI-рецепты, чат/неделя).
- **trusted** — проверенный рецепт, участвует в приоритете пула.

События «приготовил» в системе **нет**. Используются только:
- добавление в план
- удаление из плана
- замена блюда в плане
- лайк / дизлайк

---

## Когда рецепт становится trusted

Автоматически (candidate → trusted), если:
- **score ≥ 8**
- **likes ≥ 2**
- **dislikes ≤ 1**

Ожидание: нет логических ошибок, нет утечки контекста запроса, нормальный UX.

---

## Когда рецепт становится blocked

Автоматически (candidate → blocked), если:
- **dislikes ≥ 4**  
  **или**
- **score ≤ −6**

Ожидание: мусор, конфликт ингредиентов, плохие шаги — рецепт исключается из пула (generate-plan не выдаёт blocked). Один пользователь не может заблокировать рецепт одним дизлайком.

---

## Trusted safety (Stage 2.5.2)

- **trusted-рецепты не переходят в blocked автоматически.** В `recompute_recipe_score_and_trust` для `trust_level = 'trusted'` обновляется только `score`; понижение до blocked не применяется.
- **Блокировка только вручную:** `UPDATE recipes SET trust_level = 'blocked' WHERE id = ...` по-прежнему допустим (модерация, админка).

---

## Early-stage rule

- Пул на старте формируется вручную (seed/starter, ручной перевод в trusted).
- Feedback (like/dislike, план) — вспомогательный сигнал; автоматический переход candidate → trusted по правилам дополняет, а не заменяет кураторский отбор.

---

## Launch strategy (RU)

- Подготовить **20–40 trusted** рецептов перед запуском.
- Сбалансировать по приёмам: завтрак / обед / ужин / перекус (например 30% / 40% / 25% / 5% или по целевой аудитории).
- После запуска: ручная модерация раз в 2–3 дня, при необходимости пополнение trusted и блокировка проблемных.

---

## Ручная модерация

- Периодичность: раз в 2–3 дня.
- Задачи: просмотр candidate с высоким score, распределение по типам (завтрак/обед/ужин), при необходимости перевод в trusted или блокировка.

---

## Скор и сигналы

Формула score (Stage 2.5.1), clamp (Stage 2.5.2): score ограничен диапазоном **[-10, 50]**.
```
score = +2×likes − 2×dislikes + 1×added_to_plan − 0.5×replaced_in_plan − 0.5×removed_from_plan
```

- **Like/dislike:** один активный голос на (рецепт, пользователь). Повторный тот же голос не пишется; переключение like↔dislike — toggle. Спам дизлайков невозможен.
- **Free:** учитываются только `added_to_plan`, `like`, `dislike`.
- **Premium:** все сигналы (включая `replaced_in_plan`, `removed_from_plan`). План-события имеют слабый вес (0.5).

Подробности: таблица `recipe_feedback`, RPC `record_recipe_feedback`, `get_recipe_my_vote`, триггеры пересчёта при INSERT/DELETE.

---

## Подготовка пула

- Сгенерировать 30–50 рецептов в пул (candidate).
- Цель: 20–30 trusted (по правилам выше).
- Баланс по приёмам: ~30% завтрак, ~40% обед, ~30% ужин.
- Типы: курица, рыба, овощи, крупы и т.д.

---

## Таргет по trusted

- Минимум для продакшена: **25–40 trusted**.
- Для теста достаточно **15–20 trusted**.

Generate-plan приоритизирует: trusted → starter/seed → candidate, внутри группы — по `score DESC`.
