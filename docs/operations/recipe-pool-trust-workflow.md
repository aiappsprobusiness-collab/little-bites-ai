# Recipe Pool — Trust Workflow (Stage 2.5)

Операционный документ: управление качеством пула рецептов через feedback и trust level.

---

## Принцип

- **core** — вручную curated каталожные рецепты (`source = seed`); не путать с **trusted** (поведенческое доверие).
- **candidate** — черновик (новые AI-рецепты, чат/неделя).
- **trusted** — рецепт с поведенческим подтверждением (авто из candidate по правилам ниже) или ручной `manual`.

События «приготовил» в системе **нет**. Используются только:
- добавление в план
- удаление из плана
- замена блюда в плане
- лайк / дизлайк
- **shared** — пользователь успешно поделился рецептом (запись через `record_recipe_feedback`; на карточке рецепта после нативного шаринга или копирования ссылки)
- **signup_from_share** — регистрация пользователя с атрибуцией рецепту (запись через `record_recipe_feedback` из потока онбординга, когда известны `recipe_id` и `auth.uid()`)

---

## Когда рецепт становится trusted

Автоматически (candidate → trusted), если:
- **score ≥ 8**
- **likes ≥ 2**
- **dislikes ≤ 1**

Ожидание: нет логических ошибок, нет утечки контекста запроса, нормальный UX.

---

## Когда рецепт становится blocked

Автоматически (candidate → blocked), если:
- **dislikes ≥ 4**  
  **или**
- **score ≤ −6**

Ожидание: мусор, конфликт ингредиентов, плохие шаги — рецепт исключается из пула (generate-plan не выдаёт blocked). Один пользователь не может заблокировать рецепт одним дизлайком.

**Cold start (Stage 2.5.3):** рецепт не переводится в blocked, если **total_votes (like + dislike) < 3**. Новый рецепт не «умирает» от 1–2 дизлайков; даётся шанс набрать статистику. После 3+ голосов блокировка по правилам выше применяется.

---

## Trusted / core safety (Stage 2.5.2 + core)

- **trusted** и **core** не переходят в blocked автоматически. В `recompute_recipe_score_and_trust` для них обновляется только `score`; candidate-правила не применяются.
- **Блокировка только вручную:** `UPDATE recipes SET trust_level = 'blocked' WHERE id = ...` по-прежнему допустим (модерация, админка).

---

## Early-stage rule

- Каталожный пул: **seed** с `trust_level = core` (импорт curated); starter — отдельная линия без автоперевода в core в этом этапе.
- Feedback (like/dislike, план) — вспомогательный сигнал; автоматический переход candidate → trusted по правилам дополняет кураторский каталог, но **core ≠ trusted** по смыслу.

## Подбор в плане (ranking, не feedback)

- Пороги **candidate → trusted / blocked** и SQL **`recompute_recipe_score_and_trust`** при настройке composite **не менялись**.
- В **generate-plan** и в **клиентском fast-path** для выбора победителя после фильтров используются **одни и те же аддитивные бонусы** (`shared/planRankTrustShared.ts`): trust tier, вклад `recipes.score`, умеренный exploration для candidate/null, jitter. Цель — рост пула: хороший candidate может конкурировать с каталогом за счёт score и fit, без «непроницаемого» каталога.
- Рецепты с **`trust_level = blocked`** в пул подбора не попадают (Edge и клиент).

---

## Launch strategy (RU)

- Подготовить **20–40 trusted** рецептов перед запуском.
- Сбалансировать по приёмам: завтрак / обед / ужин / перекус (например 30% / 40% / 25% / 5% или по целевой аудитории).
- После запуска: ручная модерация раз в 2–3 дня, при необходимости пополнение trusted и блокировка проблемных.

---

## Ручная модерация

- Периодичность: раз в 2–3 дня.
- Задачи: просмотр candidate с высоким score, распределение по типам (завтрак/обед/ужин), при необходимости перевод в trusted или блокировка.

---

## Скор и сигналы

Формула score (Stage 2.5.1), расширение share/signup (апр. 2026), clamp (Stage 2.5.2): score ограничен диапазоном **[-10, 50]**.
```
score = +2×likes − 2×dislikes + 1×added_to_plan − 0.5×replaced_in_plan − 0.5×removed_from_plan
        + 1.5×(число событий shared) + 4×(число событий signup_from_share)
```

- **Like/dislike:** один активный голос на (рецепт, пользователь). Повторный тот же голос не пишется; переключение like↔dislike — toggle. Спам дизлайков невозможен.
- **shared / signup_from_share:** не более **одной** записи на пару (рецепт, пользователь) для каждого типа (частичные уникальные индексы в БД; RPC идемпотентен). Повторные вызовы не увеличивают score.
- **Free:** учитываются только `added_to_plan`, `like`, `dislike`.
- **Premium:** все сигналы (включая `replaced_in_plan`, `removed_from_plan`). План-события имеют слабый вес (0.5).

Пороги **candidate → trusted / blocked** и веса лайков/дизлайков/плана **не менялись**; новые сигналы только увеличивают (или через clamp ограничивают) итоговый `recipes.score`.

Подробности: таблица `recipe_feedback`, RPC `record_recipe_feedback`, `get_recipe_my_vote`, триггеры пересчёта при INSERT/DELETE.

---

## Подготовка пула

- Сгенерировать 30–50 рецептов в пул (candidate).
- Цель: 20–30 trusted (по правилам выше).
- Баланс по приёмам: ~30% завтрак, ~40% обед, ~30% ужин.
- Типы: курица, рыба, овощи, крупы и т.д.

---

## Таргет по trusted

- Минимум для продакшена: **25–40 trusted**.
- Для теста достаточно **15–20 trusted**.

Generate-plan: после фильтров слот выбирается по **composite** (slot-fit + явный бонус trust_level + вклад `recipes.score` + редкий exploration для candidate + малый jitter). Предсортировка merged пула по `trustOrder` и `score DESC` сохранена только как порядок загрузки в память.

---

## Seed pool (manual)

На старте система не имеет пользователей и голосов.

Рекомендуется:
- вручную создать **20–40 рецептов** (или импортировать сиды);
- распределить по приёмам: **breakfast**, **lunch**, **dinner**, **snack**;
- для **каталожных seed**: **trust_level = core** (см. импорт `scripts/import-infant-seed.ts` / `npm run seed:*:import` и RPC default `create_recipe_with_steps`); **trusted** — для поведенчески подтверждённых или manual, не как синоним «наш сид».

Это базовый пул, от которого строится generate-plan. Дальше feedback и авто-правила дополняют пул.

## FUTURE (not implemented)

**Trust degradation (trusted → candidate):** возможное будущее «мягкое» понижение trusted до candidate при сильном падении score и большом числе dislikes. Сейчас не реализовано; trusted не понижается автоматически.

**Plan signal limiting:** возможное ограничение накрутки: например max 1 removed_from_plan и max 1 replaced_in_plan на рецепт на пользователя в сутки. Сейчас не реализовано; план-события пишутся как история.
