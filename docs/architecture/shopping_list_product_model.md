# Список покупок: продуктовая модель (SoT)

Документ фиксирует **смысловую модель** списка продуктов (shopping list) и точки входа в UI. Детали агрегации ингредиентов — в **shopping_list_aggregation.md**.

## Модель: снимок, не зеркало плана

- Список покупок — **рабочий объект пользователя** после **явной сборки** из меню (плана).
- После сборки это **снимок (snapshot)**: изменения в `meal_plans_v2` **не** перезаписывают и не «чинят» список автоматически.
- Пользователь вручную отмечает купленное, удаляет строки, добавляет продукты; эти правки **не должны** откатываться без явного действия «собрать заново».

## Сборка из меню

1. Пользователь выбирает период: **сегодня** или **неделя** (неделя — Premium, как и доступ к списку).
2. Клиент агрегирует ингредиенты из слотов плана за период (см. `loadPlanShoppingIngredients`, нормализация в `normalizeIngredientForShopping.ts`).
3. Результат записывается в `shopping_list_items` через **полную замену строк** (`replaceItems` в `useShoppingList`), т.е. новая сборка **заменяет** предыдущее содержимое списка (осознанный снимок, не merge с черновиком). В `meta` каждой строки сохраняются **`merge_key`** (ключ агрегации) и **`source_recipes`** (рецепты-источники для фильтра по рецептам).
4. В `shopping_lists.meta` сохраняются `last_synced_range`, `last_synced_member_id`, `last_synced_plan_signature`, `last_synced_at` — чтобы сравнивать с текущим планом **без автосинхронизации**.

## Добавление из карточки рецепта (без смены модели списка)

- На экране рецепта (`RecipePage`) под списком ингредиентов — вторичная кнопка **«Добавить в покупки»** (иконка корзины). **Исключение:** для рецептов прикорма (`max_age_months < 12`, см. `isInfantRecipe` в `src/utils/infantRecipe.ts`) кнопка **не показывается** — покупки из такой карточки не предлагаем. Один активный список на пользователя: при отсутствии создаётся через `getOrCreateActiveList`; отдельный список не заводится.
- Ингредиенты переводятся в те же агрегированные позиции, что и при сборке из плана (те же ключи и единицы), с учётом выбранных **порций** на экране рецепта. **Merge:** суммирование количества при совпадении ключа; в `meta.source_recipes` добавляется рецепт (по `recipeId`, без дублей). Статус «куплено» при обновлении строки **не сбрасывается** (обновляются `amount` и `meta`).
- Фильтр по рецептам на экране списка строится по **всем** `source_recipes` в строках; рецепты, добавленные из карточки, попадают в тот же фильтр.

## Где собирать список (IA)

- **Основной вход:** экран **План** → после блока приёмов пищи за день кнопка **«Собрать список продуктов»** (secondary outline, иконка корзины). Открывается sheet **«Собрать список продуктов»** с выбором периода и **«Собрать список»**; после успеха — переход на вкладку списка в разделе избранного (`/favorites` с `state.tab = shopping_list` и при необходимости `state.shoppingListJustBuilt` для мягкого fade/slide входа панели). Одноразовая stagger-анимация строк после успешной сборки задаётся флагом в `sessionStorage` (см. `src/utils/shopping/shoppingListEntrance.ts`).
- **Верхний блок плана** — **«Собрать день»** (primary) и **«Собрать неделю»** (tertiary / текст). **«Отправить меню»** (ghost, под списком блюд, после «Собрать список продуктов») показывается только если в выбранном дне есть блюда с `recipe_id`. Неделя: пункт **«Отправить меню на неделю»** в меню «Ещё».
- **Обоснование размещения:** пользователь сначала видит блюда дня, затем логичное действие «собрать список».

## Доступ из «Избранное»

- Вкладки **Избранное** и **Мои рецепты** остаются основными чипами.
- Список покупок — **вторичный вход:** текстовая ссылка **«Покупки»** с иконкой корзины **в одной строке с полем поиска по ингредиентам** (`IngredientFilterBar.endSlot`). Это **не** точка «главного» запуска сборки; сборка позиционируется с Плана.
- На экране списка — ссылка **«К избранному и рецептам»** для возврата к рецептам.
- Premium: без подписки при открытии списка показывается paywall (как раньше).

## Строка ингредиента

- Строка строится из `getSourceRecipesFromItem` (`meta.source_recipes` или легаси `recipe_id` / `recipe_title`).
- Раскрытие по стрелке показывает список названий рецептов-источников (при активном фильтре по рецептам — только отфильтрованные; см. effective view в разделе «Фильтр по рецептам»).
- В **свёрнутой** строке, если источников больше одного, к краткому тексту позиции (название, количество) добавляется вторичная подпись вида «· в N рецепте» / «· в N рецептах» (меньший кегль, muted), чтобы было видно пересечение по рецептам без раскрытия. При одном источнике подпись не показывается.

## Фильтр по рецептам (sheet «Фильтр»)

- **Применённое состояние:** `selectedRecipeIds` в `ShoppingListView`: пустой `Set` = не фильтровать по рецептам (показать все позиции); непустой — оставить только строки, у которых `getSourceRecipesFromItem` пересекается с выбранными id.
- **Черновик в sheet:** один источник правды — `draftRecipeIds`, множество id выбранных рецептов из текущего `uniqueRecipes`. При открытии sheet черновик подставляется из applied: если фильтр не задан (`selectedRecipeIds` пуст), в черновик попадают **все** id из `uniqueRecipes` (визуально все чекбоксы включены; master «Все рецепты» отмечен). Если задан subset — копируется `selectedRecipeIds`.
- **Master «Все рецепты»:** `checked`, если в `draftRecipeIds` есть все id из `uniqueRecipes`; `indeterminate`, если выбрана только часть; снят, если ничего не выбрано. Включение master — выбрать все id; снятие — очистить `draftRecipeIds` (все конкретные чекбоксы снимаются). Отдельные рецепты **не disabled** при полном выборе: можно снять несколько ненужных без предварительного снятия master.
- **Применить:** если выбраны все рецепты из списка (или в списке нет рецептов-источников), в applied сохраняется пустой `selectedRecipeIds` (эквивалент «все рецепты»). Если выбран неполный набор — в applied сохраняется тот же набор id. Если есть хотя бы один рецепт в `uniqueRecipes`, но ни один не выбран — кнопка **Применить** неактивна (нельзя зафиксировать противоречивое состояние).
- Поиск по рецепту в sheet только сужает **видимый** список; выбор по-прежнему хранится по полному множеству id.
- **Effective view строки** при непустом `selectedRecipeIds`: не только отбор видимых позиций, но и пересчёт **количества**, **подписи «в N рецептах»** и **списка в раскрытии** по выбранным рецептам. Реализация: `computeEffectiveShoppingItemView` — сумма вкладов из `meta.source_contributions` по выбранным id и `toShoppingDisplayUnitAndAmount(meta.aggregation_unit, partialSum)`; источники в UI — пересечение `source_recipes` с фильтром. Если вкладов в meta нет (старые строки), используется **пропорция по числу рецептов** (оценка, не точная математика). После **«Собрать заново»** / новой сборки из меню вклады сохраняются в meta при записи в БД.

## Если план изменился после сборки

- Нет тревожного баннера «Меню изменилось».
- Если в списке **есть строки** и подпись плана разошлась с `last_synced_plan_signature` (или сменились период/профиль в UI) — показывается **спокойная** подсказка: «В меню есть изменения после последней сборки» и кнопка **«Собрать заново»** (и пункт меню «Собрать заново из меню»).
- Пересборка — только по явному клику; до этого список остаётся прежним снимком.

## Порции слота и подпись плана

- Порции приёма (`meals.*.servings` в `meal_plans_v2`) входят в подпись плана для списка покупок (`usePlanSignature`).
- На экране рецепта из плана (`RecipePage`) слот ищется по дате, типу приёма и **`recipe_id === id` из маршрута**. Порции из слота подставляются **как только в кэше есть** данные плана на дату (`getMealPlansByDate`), **без ожидания** загрузки объекта рецепта — иначе один кадр с начальным `1` и скачок после fetch рецепта. Если в слоте нет `servings`, дефолт берётся из рецепта после его загрузки. При обновлении с сервера без ручной правки UI подтягивает актуальные `servings`.
- Запрос плана на дату на `RecipePage` должен использовать **тот же `member_id` строки `meal_plans_v2`**, что и экран План: если из навигации не передан конкретный ребёнок, подставляется `mealPlanMemberIdForShoppingSync` (Premium «Семья» → `null`, не строка `"family"`), иначе слот не находится и порции «прыгают» между дефолтом рецепта и значением из БД.
- Тот же **queryKey** React Query, что на `MealPlanPage`: `useMealPlans` на `RecipePage` вызывается с **`starterProfile`** (аллергии/лайки/дизлайки из `useMealPlanMemberData`) и **`mutedWeekKey`** из того же правила, что начальное состояние на Плане. Иначе кэш плана на дату не совпадает, слот временно «пустой» и UI порций снова дёргается (в т.ч. у профиля взрослого).
- После сохранения порций слота мутация **`updateSlotServings`** патчит в кэше только запросы `meal_plans_v2`, чей ключ **покрывает эту `planned_date`**, затем делает **`invalidateQueries` с predicate** только по тем же ключам (не весь `meal_plans_v2` пользователя), плюс **`plan_signature`**. На **`RecipePage`** debounce-запись порций сопровождается **обязательным cleanup при unmount** (актуальные значения в refs): иначе при последнем запуске эффекта без cleanup (когда кэш уже совпал с UI) закрытие карточки до срабатывания debounce могло **не отправить** финальное сохранение в `meals.*.servings`.

## Согласование профиля с планом

- Для запросов плана и агрегации ингредиентов используется тот же выбор `member_id`, что и на экране План (в т.ч. Free + «Семья» → первый ребёнок). Утилита: `src/utils/mealPlanMemberScope.ts` (`mealPlanMemberIdForShoppingSync`).

## Что проверить вручную (dev)

1. Premium: План → после блюд «Собрать список продуктов» → сегодня/неделя → список заполнен, переход на «Покупки».
2. Изменить план (заменить блюдо) → список **не** меняется сам → появляется мягкая подсказка → «Собрать заново» обновляет.
3. Free: кнопка на Плане ведёт в paywall; «Покупки» в Избранном — то же.
4. Редактирование списка (куплено, удалить, вручную) сохраняется до явной пересборки.
5. Семья / ребёнок: список соответствует тому же плану, что виден на экране План.
6. План → карточка рецепта → сменить порции → закрыть сразу → снова открыть ту же ячейку: число порций и масштаб ингредиентов совпадают с выбранными (см. `docs/dev/plan-recipe-servings-persistence-2026-03.md`).
7. Рецепт → **«Добавить в покупки»** → toast, без автоперехода на вкладку списка; повторное добавление суммирует количества; после «Собрать из меню» добавление из рецепта **merge** без перезаписи всего списка; фильтр по рецептам показывает рецепт, добавленный из карточки.
8. Строка с **несколькими** рецептами-источниками: в свёрнутом виде видна подпись «в N рецептах»; число N совпадает со списком в раскрытии; при одном рецепте подписи нет.
9. Фильтр по рецептам: при открытии sheet при «все рецепты» все чекбоксы рецептов отмечены; снятие master снимает все; частичный выбор даёт indeterminate у master; «Применить» совпадает с отображаемым списком. Подробнее — `docs/dev/shopping-list-recipe-filter-2026-03.md`.
