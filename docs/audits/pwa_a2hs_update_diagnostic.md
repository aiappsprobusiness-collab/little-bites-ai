# PWA / A2HS диагностика

**Дата аудита:** 2025-03-14  
**Задача:** диагностика двух проблем — не показывается установка на экран телефона; после новых деплоев не видно обновления PWA. Изменения в коде в рамках этой задачи не вносились.

### 1.2. Update flow (обновление после деплоя)

| Элемент | Где реализовано |
|--------|------------------|
| **Регистрация Service Worker** | `src/main.tsx` (строки 36–71): только в **production** (`import.meta.env.PROD`), по событию `load`. Регистрация `navigator.serviceWorker.register("/sw.js", { scope: "/" })`. Результат сохраняется в `window.__swRegistration`. |
| **Тип обновления** | Не используется vite-plugin-pwa / Workbox. Кастомный SW: `public/sw.js`. Режим: **не autoUpdate в смысле «сразу активировать»** — новый worker переходит в `waiting`, пока клиент не вызовет `skipWaiting()`. То есть обновление по сути **prompt**: показываем toast «Доступна новая версия» и кнопку «Обновить», по нажатию отправляется `SKIP_WAITING` в waiting worker, затем перезагрузка. |
| **Обнаружение нового SW** | В `main.tsx`: при `reg.waiting && navigator.serviceWorker.controller` диспатчится `sw-update-available`; то же в `reg.onupdatefound` → при переходе нового worker в состояние `installed` вызывается `notifyUpdateAvailable()`. Раз в час вызывается `reg.update()` для проверки обновления SW. |
| **UI обновления** | `src/components/pwa/PWAUpdateToast.tsx`: подписка на `sw-update-available`, показ toast «Доступна новая версия» с кнопкой «Обновить»; по клику выставляется `window.__skipWaitingTriggered = true`, в `reg.waiting` отправляется `postMessage({ type: "SKIP_WAITING" })`. После `controllerchange` в `main.tsx` выполняется `window.location.reload()`. Toast не гасится по таймеру (`duration: Infinity` в Radix Root; в `use-toast.ts` пробрасывается `duration` на `Toast.Root`) — до «Обновить», свайпа в сторону или кнопки закрытия. |
| **Service Worker** | `public/sw.js`: кастомный SW без vite-plugin-pwa. Версия кэша зашита константой `CACHE_VERSION = "mom-recipes-v6"`. В `install` — precache списка статики (manifest, offline, иконки). В `activate` — удаление старых кэшей, `clients.claim()`. В `fetch`: для навигации — network-first с offline fallback; для `sw.js` и `manifest.json` — **не перехватываются** (всегда сеть); для API/Supabase — не перехватываются; для same-origin ассетов — stale-while-revalidate. В `message` — обработка `SKIP_WAITING` и вызов `self.skipWaiting()`. |

**Итог по update flow:**  
Регистрация и логика «новый worker → toast → Обновить → skipWaiting → reload» реализованы.

**Исправление (после аудита):** В `public/sw.js` константа `CACHE_VERSION` заменена на плейсхолдер `__APP_BUILD_VERSION__`; плагин в `vite.config.ts` при сборке подставляет версию билда (git short hash + timestamp). Таким образом, после каждого production build содержимое `dist/sw.js` меняется, браузер видит новый worker и срабатывает логика обновления.

### 1.3. Обновление install flow (2025-03): ценностные триггеры и fallback

После доработки PWA install UX добавлено:

- **Новые источники триггеров (trigger sources):**
  - **После 2 успешно полученных рецептов** — в `ChatPage.tsx` ведётся счётчик `a2hs_recipe_success_count` (localStorage); при достижении 2 диспатчится `A2HS_EVENT_AFTER_TWO_RECIPES` (флаг `a2hs_two_recipes_dispatched`).
  - **После первого успешно собранного дня** — в `MealPlanPage.tsx` при завершении генерации плана на день диспатчится `A2HS_EVENT_AFTER_FIRST_DAY` (флаг `a2hs_first_day_dispatched`).
  - **После первой успешно собранной недели** — при завершении плана на неделю диспатчится `A2HS_EVENT_AFTER_FIRST_WEEK` (флаг `a2hs_first_week_dispatched`).
  - Сохранены legacy: `A2HS_EVENT_AFTER_FIRST_RECIPE` (первый рецепт), `A2HS_EVENT_AFTER_FIRST_PLAN` (для обратной совместимости, из MealPlan больше не диспатчится).

- **Тексты install prompt** зависят от триггера и типа подписки (`PWAInstall.tsx` + `useSubscription().hasAccess`):
  - Free: без упоминания списка продуктов («рецепты и меню» / «меню будет всегда под рукой»).
  - Premium/Trial + триггер «неделя»: «Меню и список продуктов уже готовы. Добавьте приложение на экран, чтобы всё было под рукой.»
  - Рецепты: «Добавьте приложение на экран — рецепты будут открываться как обычное приложение.»

- **Ручной fallback в Профиле:** блок «Приложение» показывается всегда при `!isInstalled && !isStandalone()`. Если системный `beforeinstallprompt` недоступен: на iOS — диалог «Поделиться → На экран Домой»; на Android/других — диалог «Меню браузера → Установить приложение / Добавить на главный экран».

- **Антиспам:** прежняя логика suppression/cooldown не менялась; один триггер на сессию (`a2hs_trigger_source`), не показывать в standalone и после установки.

## 2.1. Конкретная UX-логика показа установки (по коду)

- **Когда именно должен показываться install prompt (модалка)?**  
  Не «сразу» и не по таймеру. Только после **одного из ценностных событий** (см. разд. 1.3): первый рецепт, 2 рецепта, первый день плана, первая неделя плана (или legacy FIRST_PLAN). После диспатча события — **задержка 8–15 сек**, затем `setShowModal(true)`.

- **Где зашито:**  
  `src/hooks/usePWAInstall.ts`: подписка на события `A2HS_EVENT_AFTER_FIRST_PLAN`, `A2HS_EVENT_AFTER_FIRST_RECIPE`, `A2HS_EVENT_AFTER_TWO_RECIPES`, `A2HS_EVENT_AFTER_FIRST_DAY`, `A2HS_EVENT_AFTER_FIRST_WEEK`; вызов `scheduleShow(trigger)` с задержкой; при показе текст в модалке по `installPromptTriggerSource` и подписке (Free/Premium).

- **Какие флаги отключают показ навсегда или временно:**  
  `a2hs_dismissed_forever === "1"` — навсегда; `a2hs_attempt_count >= 3` — навсегда; `a2hs_next_eligible_at` в будущем — до истечения cooldown (3 или 7 дней); `a2hs_trigger_source !== ""` — триггер уже использован, повторные события не планируют модалку.

- **Может ли prompt быть уже подавлен и поэтому не показываться?**  
  Да. Один раз «Позже» → 3 дня; два раза → 7 дней; три раза или «Установить» → навсегда. Плюс если пользователь никогда не генерировал план и не получал рецепт — модалка не запланируется.

- **Показ только на Android Chrome?**  
  Модалка планируется и на Android, и на iOS (при `hasA2HSSupport`). Но на Android кнопка «Установить» активна только при `deferredPrompt` (т.е. при срабатывании `beforeinstallprompt`). Если пользователь тестирует на iOS — нативного prompt нет, будет только наша инструкция в модалке и ручная кнопка в профиле.

- **Ручная кнопка «Установить приложение» в профиле:**  
  Секция «Приложение» показывается всегда при не установленном и не standalone. По клику: при наличии deferred prompt — системная установка; на iOS — диалог «Поделиться → На экран Домой»; на Android/других без prompt — диалог с инструкцией (меню браузера → Установить приложение / Добавить на главный экран).

## 3. Почему не показывается установка

### 3.1. Вероятные причины (по приоритету)

1. **Нет иконки 192×192 в манифесте (критично для Android Chrome)**  
   В `public/manifest.json` в `icons` указаны только размеры **512×512** и **1024×1024**. По [критериям установки Chrome](https://web.dev/articles/install-criteria) для срабатывания installability и события `beforeinstallprompt` в манифесте должны быть **иконка 192×192 и иконка 512×512**. При отсутствии 192px Chrome может не считать приложение устанавливаемым и **не вызывать `beforeinstallprompt`**. В результате `deferredPrompt` в приложении остаётся `null`, системный баннер не показывается (мы его перехватываем), кастомная кнопка в профиле не показывается (`hasInstallOption` зависит от `deferredPrompt` на Android), а модалка либо не планируется (нет `hasA2HSSupport` без iOS), либо показывается без рабочей кнопки «Установить» (только «Позже» и текст для iOS).

2. **Модалка показывается только после первого плана или первого рецепта и с задержкой 8–15 сек**  
   Пользователь может не дойти до генерации плана или рецепта, либо закрыть вкладку до истечения задержки. Тогда модалка установки не появится вообще. Это не баг, а заложенное поведение: авто-подсказка привязана к одному из двух событий и отложена.

3. **Suppression: один раз «Позже» или «навсегда»**  
   После первого «Позже» — cooldown 3 дня; после второго — 7 дней; после третьего — `a2hs_dismissed_forever === "1"`, модалка больше не показывается. Если пользователь ранее отклонил подсказку (или тестировщик), повторно авто-модалка не появится.

4. **Триггер уже «потрачен»**  
   Если хотя бы раз был запланирован показ по плану или рецепту (`a2hs_trigger_source` в localStorage уже установлен), новые события `A2HS_EVENT_AFTER_FIRST_PLAN` / `A2HS_EVENT_AFTER_FIRST_RECIPE` не запускают повторное планирование. То есть повторный заход «после первого рецепта» не откроет модалку снова.

5. **Только для авторизованных**  
   В `usePWAInstall` планирование модалки выполняется только при наличии `user`. Неавторизованный пользователь не увидит авто-модалку (ручная кнопка в профиле ему недоступна по роутингу).

### 3.2. По платформам

- **Android (Chrome и др.):**  
  Главная техническая причина — отсутствие иконки 192×192 в манифесте, из‑за чего браузер может не вызывать `beforeinstallprompt`. Тогда нет ни системного баннера, ни нашей кнопки в профиле (она показывается только при `hasInstallOption` = true, что на Android требует `deferredPrompt`). Модалка может показаться только если по какой-то причине `beforeinstallprompt` всё же сработал (например, в другом браузере/среде), при этом сработают те же ограничения по триггеру, задержке и suppression.

- **iPhone / iOS Safari:**  
  На iOS событие **`beforeinstallprompt` не поддерживается** — это ожидаемое поведение платформы. У нас для iOS: `hasA2HSSupport` возвращает true за счёт `isIOS()`, поэтому в профиле показывается секция «Установить приложение» и при нажатии — диалог с инструкцией «Поделиться → На экран Домой». Авто-модалка тоже может запланироваться (по первому плану/рецепту), но кнопка «Установить» в ней не будет активна (`canInstall` = false), будет только текст про Safari и кнопка «Позже». Итого: на iOS пользователь может установить приложение только через ручную кнопку в профиле и нашу инструкцию; отдельной «добавить на экран» в системном UI у Safari нет — это ограничение iOS, не баг приложения.

## 4.1. Почему не показывается обновление после нового деплоя

**Причина:** между деплоями фронта **файл `sw.js` не меняется**.

- `public/sw.js` копируется в `dist` как есть. В нём зашита константа `CACHE_VERSION = "mom-recipes-v6"`, не привязанная к хешу билда или дате.
- Браузер при проверке обновления SW сравнивает новый ответ от сервера с текущим зарегистрированным скриптом. Если содержимое совпадает (byte-for-byte), новый worker не устанавливается, `reg.installing`/`reg.waiting` не появляются.
- В результате не вызывается `notifyUpdateAvailable()`, не диспатчится `sw-update-available`, пользователь не видит toast «Доступна новая версия» и не получает кнопку «Обновить».
- Фактически установленный SW продолжает обслуживать страницу старым кэшем (в т.ч. старые JS/CSS по stale-while-revalidate), пока пользователь не сделает жёсткое обновление или не очистит кэш. Это **ошибка конфигурации/подхода**: версионирование SW не привязано к деплою, поэтому механизм «новый деплой → новый SW → toast обновления» не срабатывает.

Дополнительно: даже если бы `sw.js` менялся при каждом деплое, навигация у нас обрабатывается по схеме network-first, то есть `index.html` запрашивается из сети. Но до обновления SW клиент продолжает работать со старым worker’ом, который отдаёт старые ассеты из кэша, поэтому без смены версии SW пользователь остаётся на старом фронте.

## 5. Что именно сломано / чего не хватает

### По приоритету

1. **Манифест: нет иконки 192×192** — приложение не проходит критерии установки Chrome, `beforeinstallprompt` скорее всего не вызывается. Это основная причина, почему пользователь не видит ни системное предложение, ни нашу кнопку на Android.
2. **Service Worker не версионируется при деплое** — содержимое `sw.js` статично, браузер не видит «новую версию» после выката, update flow (toast «Доступна новая версия») фактически не срабатывает.
3. **Авто-модалка установки жёстко привязана к первому плану/рецепту и задержке 8–15 сек** — пользователь может никогда не попасть в этот сценарий или не дождаться; нет запасного сценария (например, показ после N визитов или на определённом экране без привязки к первому рецепту).
4. **Ручная кнопка в профиле на Android бесполезна без `beforeinstallprompt`** — секция «Установить приложение» показывается при `hasInstallOption`; на Android это true только при наличии `deferredPrompt`, которого не будет, если нет 192px иконки. Получается замкнутый круг.
5. **Хук `useA2HSInstall` не используется** — мёртвый код; если планировался ручной install на landing/auth, он не подключён.
6. **Нет отдельной документации/подсказки для iOS** в самом приложении (кроме диалога в профиле) — на iOS установка только через инструкцию в профиле; это ограничение платформы, но можно явно описать в интерфейсе (например, на welcome/prelogin).

### Ожидаемое поведение vs наши ошибки

| Ситуация | Ожидаемое поведение / ограничение | Наш баг / конфиг |
|----------|-----------------------------------|------------------|
| Нет 192px в manifest | Chrome не считает PWA устанавливаемым, не шлёт `beforeinstallprompt` | Ошибка конфигурации манифеста |
| iOS без beforeinstallprompt | Только ручная установка через Share → Add to Home Screen | Нет бага; инструкция есть в профиле |
| Модалка только после первого плана/рецепта | Задумка продукта | Не баг, но узкий сценарий |
| После деплоя нет обновления SW | Браузер обновляет SW только при изменении скрипта | Ошибка: sw.js не версионируется под деплой |
| Suppression после 3× «Позже» | Задумка, чтобы не надоедать | Ожидаемо |
