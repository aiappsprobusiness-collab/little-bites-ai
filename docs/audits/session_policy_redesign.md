# Redesign session policy audit

**Дата:** 2025-03-14  
**Задача:** аудит текущей auth/session/device-логики и проектирование новой политики сессий без изменения кода.

**Цель:** разрешить нормальный сценарий «PWA + браузер на одном телефоне» без раздражающего разлогина, сохранив защиту от шаринга аккаунта на 5+ устройств.

---

## 1. Как система работает сейчас

### 1.1. Auth / session flow

- **Supabase Auth:** используется стандартный клиент с `localStorage` как хранилищем сессии, `persistSession: true`, `autoRefreshToken: true`, `detectSessionInUrl: true` (`src/integrations/supabase/client.ts`).
- **Вход:** по паролю — `useAuth.signIn` → `signInWithPassword` → при успехе вызывается `setActiveSessionKeyForUser(userId)`. По magic link / email confirmation — `AuthCallbackPage` после получения сессии вызывает `setActiveSessionKeyForUser(userId)` перед редиректом.
- **Выход:** `useAuth.signOut` очищает локальный ключ сессии и вызывает `supabase.auth.signOut()`.
- **Проверка сессии:** выполняется в `useAuth`:
  - при старте (после `getSession()`);
  - при каждом `onAuthStateChange`;
  - при возврате в приложение (`visibilitychange` → снова `validateActiveSession`).

Источник истины по сценарию описан в `docs/decisions/SINGLE_ACTIVE_SESSION.md`.

### 1.2. Device / session logic (anti-sharing)

- **Одна активная сессия на аккаунт.** В БД в `profiles_v2.active_session_key` хранится один текстовый ключ. При каждом новом логине генерируется новый случайный ключ (24 байта, hex), записывается в БД и в `localStorage` под ключом `lb_active_session_key` (`src/utils/activeSessionKey.ts`).
- **Валидация:** клиент запрашивает `profiles_v2.active_session_key` по `user_id` и сравнивает с `localStorage`. Если ключ в БД отсутствует (legacy) — текущий клиент считается активным, ключ записывается. Если ключ в БД не совпадает с локальным (или локального нет при наличии в БД) — сессия считается заменённой другим устройством: выполняется `setSessionInvalidReason(SESSION_INVALID_REASON_REPLACED)`, `clearStoredSessionKey()`, `supabase.auth.signOut()`.
- **Идентификация «устройства»:** явного `device_id` или fingerprint в коде нет. «Устройством» фактически является **контекст хранилища**: один origin + один partition. У каждого такого контекста свой `localStorage`, значит свой экземпляр ключа `lb_active_session_key` и свой набор Supabase auth-токенов.

### 1.3. Где заложена жёсткая реакция

- **При логине:** `setActiveSessionKeyForUser` всегда делает **update** одного поля `active_session_key` в `profiles_v2` по `user_id`. Предыдущее значение перезаписывается. Все остальные клиенты (любые другие контексты хранения) при следующей проверке увидят несовпадение и будут разлогинены.
- **При проверке:** `validateActiveSession` в `useAuth` при `valid: false` и `reason === SESSION_INVALID_REASON_REPLACED` немедленно вызывает signOut и очистку ключа. Никакой задержки, никакого учёта «то же устройство / PWA vs browser» нет.
- **Edge Functions и RLS:** логику активной сессии не используют. Проверка выполняется только на клиенте и при обращении к `profiles_v2` (чтение `active_session_key`). Бэкенд не инвалидирует токены Supabase по ключу сессии.

### 1.4. Трактование PWA и браузера на одном телефоне

- **Считаются ли разными устройствами?** Да. Причина — **раздельное хранилище**.
  - В типичной конфигурации PWA, открытая с главного экрана (standalone), работает в отдельном browsing context: тот же origin, но **другой storage partition** (свои `localStorage`/`sessionStorage`). Обычная вкладка браузера имеет свой partition.
  - Итог: у PWA и у браузера — разные `localStorage`, значит разные `lb_active_session_key` и разные Supabase auth-токены. С точки зрения текущей логики это **два разных «устройства»**.
- **На основании чего:** отдельный storage context (и как следствие — отдельный ключ в localStorage). Ни `display-mode`, ни `device_id`, ни fingerprint в session/device-логике не используются; `isStandalone()` в проекте используется только для UI (скрытие кнопки установки PWA, секция «Приложение» в профиле), не для auth.
- **Что вызывает разлогин:** при входе пользователя в браузере вызывается `setActiveSessionKeyForUser` → в БД записывается новый ключ, в браузере — в localStorage. У PWA остаётся старый ключ в своём localStorage. При следующем открытии PWA или при срабатывании `visibilitychange` в PWA вызывается `validateActiveSession` → `localKey !== serverKey` → signOut. То есть разлогин инициируется на фронте из-за несовпадения ключа; backend явно сессию не убивает (Supabase refresh token остаётся валидным до явного signOut или истечения).

## 2. Почему текущая схема раздражает нормального пользователя

- **Конкретный сценарий:** пользователь установил PWA на телефон, залогинился в PWA. Потом открыл тот же сайт в обычном браузере на том же телефоне и снова вошёл (например, перейдя по ссылке из письма или для удобства). Текущая логика трактует браузер как «новое устройство», перезаписывает `active_session_key`. При следующем открытии PWA проверка видит несовпадение ключа и выполняет принудительный выход. Пользователь видит переход на `/auth` и toast: «Аккаунт открыт на другом устройстве. Войдите снова.» — при том что «другое устройство» это он же на том же телефоне.
- **Почему это не равно реальному шарингу:** один человек на одном телефоне использует два контекста (PWA и браузер) — типичный паттерн (ссылки в мессенджерах открываются в браузере, основное использование — в приложении). Жёсткая политика «одна сессия = один ключ в БД» не различает «два контекста на одном устройстве» и «два разных физических устройства», что и даёт ложное срабатывание anti-sharing для нормального использования.

## 3. Какие риски есть, если просто убрать ограничения

- **Сценарии злоупотребления:** один платящий аккаунт может быть использован одновременно на многих устройствах (семья/друзья, «общий пароль»). Это снижает выручку и искажает метрики использования.
- **Почему нельзя полностью снимать защиту:** продукт с подпиской и лимитами (daily_limit, premium-фичи) предполагает привязку к одному пользователю/дому. Полное отсутствие ограничения по устройствам/сессиям превращает подписку в легко шарящийся ресурс на 5+ человек без дополнительной оплаты. Нужна политика, которая ограничивает именно **явный мультидевайсный шеринг**, а не смену контейнера (PWA ↔ браузер) на одном устройстве.

## 4. Рекомендуемая новая session policy

### 4.1. Целевая модель

- **Принцип:** не наказывать пользователя за смену контейнера на том же устройстве; ограничивать явный мультидевайсный шеринг.
- **Разрешать:** один пользователь на одном телефоне — PWA и браузер одновременно без принудительного разлогина при переключении между ними; ограниченное число активных сессий/контекстов (см. лимиты ниже).
- **Ограничивать:** 4–5+ одновременных «устройств» (в смысле независимых контекстов/сессий); подозрительную мультидевайсную активность; явный account sharing.

### 4.2. Лимиты активных сессий / устройств

Рекомендуемый вариант:

- **Free:** до **2** активных клиентских контекстов (сессий).
- **Premium / Trial:** до **3** активных клиентских контекстов.

Обоснование:

- Два контекста для free покрывают типичный кейс «PWA + браузер на одном телефоне» и, при желании, «телефон + один другой девайс» (например ноутбук). Этого достаточно для нормального использования без ощущения «выкинуло ни за что».
- Три для premium дают запас для «телефон PWA + телефон браузер + планшет/ноут» без поощрения шаринга на 5+ человек. Жёсткий лимит «1» для всех — как сейчас — даёт плохой UX при PWA+браузер; лимиты 3/4 и выше без разделения free/premium усиливают риск шеринга на free.

Альтернатива (все тарифы по 2 сессии) проще в реализации, но менее гибка для платящих пользователей; при необходимости её можно принять как первый шаг.

### 4.3. Отношение к PWA + browser на одном телефоне

Рекомендация: **считать их отдельными сессиями, но допускать в рамках одной «device group» или в рамках лимита без немедленного logout.**

- **Не делать:** «склеивание» storage между PWA и браузером — технически ненадёжно (разные partitions, нет стандартного API для общего хранилища между standalone и tab).
- **Делать:** смягчить политику так, чтобы два контекста на одном устройстве не трактовались как повод для немедленной инвалидации одной из сессий. Конкретно:
  - Хранить **не один ключ**, а **список активных сессий** (или связку session_key + опциональные метаданные). Каждый логин добавляет/обновляет свою запись; при превышении лимита деактивируется самая старая или наименее активная сессия, а не «все кроме последней».
  - Для мягкого распознавания «тот же девайс» (PWA + browser) использовать эвристики по сигналам (см. раздел 5): например, одна и та же пара (user_agent, разрешение экрана, время последней активности в узком окне) может учитываться при приоритизации какой сессию вытеснить, или при показе сообщения «достигнут лимит устройств» вместо немедленного logout текущего клиента. Желательно не строить критичную логику только на одном сигнале (например только display-mode), чтобы не поломаться при изменениях браузеров.

Итог: PWA и браузер остаются двумя разными сессиями (два ключа в списке), но оба могут оставаться валидными, пока не превышен лимит сессий. При превышении лимита — вытесняем по правилам (старая/наименее активная), а не «последний логин выбивает все остальные».

### 4.4. Правила soft / hard enforcement

- **Soft:** при достижении лимита сессий не выполнять немедленный signOut на текущем клиенте; показывать сообщение о достижении лимита и предложить выйти на другом устройстве или в настройках завершить старую сессию. Новый логин с «лишнего» контекста может либо отклонить вход с текстом «Достигнут лимит активных устройств», либо вытеснить самую старую сессию (её при следующей проверке разлогинит).
- **Hard:** при явных признаках abuse (например множество разных сессий за короткое время с сильно различающимися UA/разрешениями) можно дополнительно ужесточать (например временно снижать лимит или требовать повторный вход). Это можно ввести на втором этапе после базовой политики по лимиту сессий.

## 5. Какие сигналы использовать

Оценка применимости сигналов для различения «один пользователь, один девайс» vs «шаринг»:

| Сигнал | Плюсы | Минусы / риски | Применение |
|--------|--------|-----------------|------------|
| **user_agent** | Доступен везде, хорошо различает девайсы/ОС | Легко подделывается; один телефон даёт один UA и для PWA, и для браузера | Группировка «похожих» сессий (одно устройство); при вытеснении — не вытеснять сессию с тем же UA+разрешением, если есть «чужие» |
| **platform / OS** | Стабилен в рамках одного устройства | Тот же для PWA и браузера на одном телефоне | Поддержка эвристики «тот же девайс» |
| **display_mode (standalone / browser)** | Прямо указывает на PWA vs браузер | Не во всех браузерах одинаково; нужна осторожность | Опционально: при равном приоритете вытеснения предпочитать сохранять сессию с тем же display_mode, что и текущая активность, или учитывать при «device group» |
| **last_seen / last_activity** | Надёжный критерий «самая старая» сессия | Требует обновления с бэкенда или из клиента | Основной критерий вытеснения при превышении лимита |
| **Близость логинов по времени** | Два входа подряд с одного устройства (PWA→browser) часто идут с разницей в минуты | Ложные срабатывания при быстром переключении между реальными устройствами | Мягкий сигнал: не считать «подозрительным» два контекста с близким last_seen и похожим UA |
| **IP / geo** | Сильно различает географически разные устройства | Не всегда есть на клиенте; нужен бэкенд; VPN/мобильный интернет меняются | По желанию на втором этапе для детекции аномалий; не обязательно для первой версии |
| **fingerprint / client_id** | Может стабильно идентифицировать устройство | Сложность, приватность, нестабильность на мобильных | Не рекомендовано в первой итерации; при введении — только как дополнительный мягкий сигнал |
| **Количество активных контекстов** | Простой и надёжный | — | Основа лимита: хранить N последних сессий и при N+1 вытеснять по last_seen/активности |
| **Параллельная активность** | Много запросов с разных сессий одновременно — признак шаринга | Нужна инфраструктура подсчёта | Второй этап, опционально |

Рекомендация для первой версии:

- Использовать **список сессий с last_seen** и **лимит по количеству** (2/3). При превышении — вытеснять сессию с самым старым last_seen (или наименее активную).
- Опционально сохранять при логине **user_agent** (и при желании **display_mode**) в метаданных сессии — не для жёсткого решения «то же устройство», а чтобы в UI показывать «телефон», «браузер» и при вытеснении по возможности не выбивать сессию с тем же UA (если есть другие кандидаты). Это уменьшает вероятность, что пользователя выкинет из PWA сразу после входа в браузере на том же телефоне, если лимит 2 и одна из двух сессий — «старая» на другом устройстве.

## 6. UX при конфликте / превышении лимита

- **Что видит пользователь:**
  - При достижении лимита устройств и попытке входа с нового контекста: сообщение «Достигнут лимит активных устройств (N). Выйдите на одном из них или завершите сессию в настройках профиля», без разлогина уже открытых сессий (если текущий клиент уже в лимите).
  - Если решено «новый логин вытесняет самую старую сессию»: на вытесненном клиенте при следующей проверке — не агрессивный «Аккаунт открыт на другом устройстве», а нейтральное «Сессия на этом устройстве завершена из-за входа с другого. Войдите снова при необходимости» и редирект на `/auth`.
  - В профиле: раздел «Активные сессии» (опционально на втором этапе) — список с примерным описанием (браузер/телефон, последняя активность), кнопка «Завершить сессию» для выбранной. Это даёт контроль и снижает ощущение «меня выкинуло без спроса».

- **Как не бесить нормального пользователя:** не делать единственной активной сессией «последний вошедший»; разрешить 2 (free) и 3 (premium) сессии; при вытеснении показывать понятное объяснение и не смешивать с «кто-то украл аккаунт»; по возможности не вытеснять сессию с тем же устройством (по UA/display_mode), если есть другие кандидаты.

- **Как сдерживать шеринг:** жёсткий лимит 2/3 сессий; при необходимости — логирование и последующий анализ аномалий (много смен сессий, сильно разные UA/geo) для ужесточения политики точечно.

## 7. Что нужно менять в коде

Ниже — список областей без немедленного внесения изменений (только план).

### 7.1. База данных

- **profiles_v2:** вместо одного поля `active_session_key` (text) ввести структуру для нескольких сессий. Варианты:
  - Новая таблица `user_sessions` (user_id, session_key, last_seen, meta jsonb с user_agent/display_mode и т.д.), уникальность по (user_id, session_key) или по id; RLS по user_id. Или хранить в `profiles_v2` колонку типа jsonb `active_sessions` вида `[{ "key": "...", "last_seen": "...", "ua": "..." }]` с ограничением по длине (2 или 3). В обоих случаях при логине добавлять/обновлять запись, при превышении лимита удалять/деактивировать самую старую.
- Миграция: добавить новую таблицу или колонку, перенести существующую логику на «множественные сессии», затем (в другой миграции) удалить `active_session_key` после переключения фронта и бэкенда.

### 7.2. Backend (Edge / RPC)

- При необходимости — RPC или Edge для «добавить/обновить сессию» и «получить список сессий» с учётом лимита и last_seen. Логика вытеснения (какую сессию удалить при превышении лимита) может быть в БД (триггер/функция) или в Edge. Проверка «текущий ключ в списке валидных» — при запросе профиля или отдельным эндпоинтом.

### 7.3. Фронт

- **activeSessionKey.ts:** заменить «один ключ» на работу со списком: при логине — вызов API «зарегистрировать/обновить сессию» (передача ключа, опционально user_agent/display_mode), получение ответа «ок» или «лимит, вытеснена сессия X». Хранение в localStorage по-прежнему своего session_key для сравнения с ответом сервера (текущий ключ в списке активных и не истёкших).
- **useAuth.tsx:** после логина вызывать новый метод «зарегистрировать сессию» вместо `setActiveSessionKeyForUser`. В `validateActiveSession` запрашивать не один ключ из profiles_v2, а список активных сессий и проверять, что локальный ключ есть в списке и не помечен как вытесненный. При невалидной сессии различать причину: «вытеснена из-за лимита» vs «явный logout» — и по возможности писать в sessionStorage разный текст для AuthPage.
- **AuthPage.tsx:** добавить отображение сообщения для «сессия завершена из-за лимита устройств» (если введётся отдельная причина).
- **AuthCallbackPage.tsx:** после установки сессии вызывать ту же новую логику регистрации сессии (как при логине по паролю).
- **Профиль (опционально):** экран «Активные сессии» с возможностью завершить выбранную — требует API списка сессий и «удалить сессию по key».

### 7.4. Документация

- **docs/decisions/SINGLE_ACTIVE_SESSION.md:** переименовать или заменить на документ с новой политикой (несколько сессий, лимиты, PWA+browser, вытеснение).
- **docs/database/DATABASE_SCHEMA.md:** описать новую структуру (таблица или колонка сессий), убрать/обновить описание `active_session_key`.

## 8. План внедрения по шагам

1. **Минимальный безопасный вариант (первый шаг)**  
   - Ввести структуру «несколько сессий» в БД (таблица или jsonb) с лимитом 2 для всех.  
   - При логине: добавлять сессию; при превышении лимита — вытеснять самую старую по last_seen.  
   - На клиенте: проверять «мой ключ в списке»; при отсутствии — logout с причиной «вытеснена» или «заменена».  
   - Сообщения: «Достигнут лимит устройств» при отказе во входе; «Сессия на этом устройстве завершена…» на вытесненном клиенте.  
   - Не трогать пока UA/display_mode и «device group» — только лимит и список сессий.

2. **Улучшенный вариант (второй шаг)**  
   - Разделить лимиты free (2) и premium/trial (3).  
   - Сохранять при логине user_agent (и опционально display_mode); при вытеснении предпочитать удалять сессию с другим UA (если есть), чтобы реже выбивать PWA при входе в браузере на том же телефоне.  
   - В профиле: экран «Активные сессии» с возможностью завершить сессию.

3. **Приоритеты**  
   - Критично: шаг 1 (лимит 2 сессии, вытеснение по last_seen) — решает проблему PWA+браузер без усложнения.  
   - Далее: разделение free/premium и мягкая эвристика по UA/display_mode.  
   - По желанию: управление сессиями в профиле, затем мониторинг аномалий и ужесточение.

## Ответ на ключевой вопрос

**Как разрешить нормальный сценарий «PWA + browser на одном телефоне» и при этом не превратить аккаунт в легко шарящуюся подписку на 5 устройств?**

- **Разрешение сценария PWA + browser:** перестать хранить один глобальный `active_session_key` и заменить его на **список активных сессий** с лимитом (2 для free, 3 для premium). Тогда вход в браузере не перезаписывает ключ PWA, а добавляет вторую сессию; обе остаются валидными, пока в списке не больше лимита. Таким образом, один пользователь на одном телефоне может спокойно использовать и PWA, и браузер без принудительного выхода.
- **Защита от шаринга на 5+ устройств:** жёсткий лимит числа активных сессий (2 или 3). При попытке входа с нового контекста сверх лимита — либо отказ во входе с сообщением «Достигнут лимит устройств», либо вытеснение самой старой/наименее активной сессии (и тогда только один клиент получит выход при следующей проверке). Так аккаунт не может одновременно быть активен на 5+ независимых контекстах.
- **Баланс:** два контекста (PWA и браузер) на одном телефоне укладываются в лимит 2; третий контекст (например второй телефон или ноутбук) тоже допустим при лимите 3 для premium. Реальный шаринг на 4–5 человек потребует постоянного вытеснения сессий и неудобства («меня выкинуло»), что сдерживает злоупотребление при сохранении нормального UX для одного пользователя.
