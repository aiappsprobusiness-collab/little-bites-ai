# Аудит пула рецептов 12–36 месяцев (качество недельного плана)

**Дата аудита:** 2026-03-29  
**Источник данных:** сид-каталог `data/toddler-seed/toddler-catalog-recipes.json` (основной курируемый набор 12–36 мес в репозитории).  
**Повторный запуск метрик:** `npx tsx scripts/audit-recipe-pool-12-36.ts` (скрипт не меняет генератор плана).

## Контекст проблемы

- В проде пользователь может видеть **~100** рецептов в диапазоне 12–36 мес (сид + `starter` / `manual` / `week_ai` / `chat_ai`). В репозитории **базовый курируемый** набор сейчас **84** строки в JSON — расхождение нормально, если в БД есть доп. рецепты.
- Уже включён **ingredient diversity** (`shared/keyIngredientSignals.ts`), но при **малом числе кандидатов на слот** и **перекосе primary_key** алгоритм только мягко штрафует повторы; «разнообразие» ограничено **составом пула**.
- Цель этого документа — отделить **дефицит данных** от **дефицита алгоритма**.

## 2. Эффективный пул после типичных фильтров (модель)

Скрипт имитирует цепочку, близкую к **generate-plan** для ребёнка 12+:

- `min_age_months` / `max_age_months`;
- keyword-guards (`TODDLER_UNDER_24_*` до 24 мес и т.д.);
- совпадение `meal_type` со слотом;
- **обед:** только `is_soup` или «суп» в названии (как на Edge);
- **завтрак:** не суп;
- sanity по слоту (перекус на обед/ужин, «тяжёлый» обед на завтрак);
- **упрощённые** сценарии аллергий/dislikes (для точных цифр по аллергиям в проде используйте полный `allergenTokens`).

## 3. Достаточность пула (вывод)

### 3.1 Хватает ли ~84–100 рецептов на 7×4 = 28 слотов?

**Формально** слоты можно заполнить: кандидатов на слот при отсутствии фильтров **15–25**.  
**Качественно** — **недостаточно** для стабильного разнообразия по `primary_key`:

- При **18** уникальных primary на **84** рецепта средняя «ёмкость» на ключ ~4–5 блюд; за неделю с окном +4 дня и штрафами повторы **rice / apple / chicken / turkey / cottage_cheese** неизбежны.
- **Snack / breakfast** — узкие по смыслу (много каш, творога, яблок/бананов): даже с diversity **альтернатив мало**.

### 3.2 Оценка целевого размера (эвристика, не SQL)

| Уровень | Рецептов (12–36, все источники пула) | Комментарий |
|---------|--------------------------------------|-------------|
| **Минимум** | **180–220** | Запас по слотам после lunch=soup и типичных exclude title/id; снижение коллапса primary |
| **Комфорт** | **280–350** | Запас под аллергии/dislikes и под `POOL_LIMIT_*` + старые `created_at` в выборке |
| **С запасом под A/B** | **400+** | Если нужны региональные/культурные ветки без потери возрастных правил |

**Ориентир по повторам:** чтобы держать большинство ключей **≤2–3 раза за 7 дней + окно**, ориентировочно нужно **не меньше ~2× числа «логических слотов»** уникальных блюд с **разными primary** на слот — на практике это приводит к оценкам выше, а не к «ещё 20 рецептов».

### 3.3 Узкие места по `meal_type`

1. **Lunch** — после фильтра «только суп» остаётся **~77%** от обедов в каталоге; при dislikes по овощам срез **катастрофический** (пример с морковью: **2**).
2. **Snack** — мало ключей кроме **cottage_cheese / apple / pumpkin / yogurt / banana**.
3. **Breakfast** — много **oatmeal / cottage_cheese / rice / яйца**; при аллергии на яйцо половина завтраков пропадает.

## 4. План расширения пула (структурно)

### 4.1 Сколько добавить по `meal_type` (от текущих 25/22/19/18)

Ориентир **к следующему комфортному уровню** (+100…+150 к курируемому сиду):

| meal_type | Добавить (ориентир) | Зачем |
|-----------|---------------------|--------|
| breakfast | **+18…+25** | Снизить долю одной каши/творога; больше безъяичных белковых завтраков |
| lunch | **+12…+18** | Новые **супы** с базой не «курица+рис+морковь»: рыба, бобовые, другие овощи |
| snack | **+15…+22** | Меньше яблоко/банан/творог как единственный «якорь» |
| dinner | **+15…+22** | Рыба, бобовые, индейка/говядина не только в связке с рисом/картофелем |

### 4.2 Снизить долю (через новые рецепты с другим первым ключом)

- **apple, banana, rice, oatmeal, pumpkin, potato** — не удаляя старые, **разбавлять** блюдами, где первый распознанный ключ — другой продукт.
- **chicken / turkey** — частично замещать **рыбой**, **бобовыми**, **творогом+овощ** в другом порядке ингредиентов (чтобы сменился `primary_key`).

### 4.3 Добавить продукты / категории

- **Рыба:** лосось/сёмга, треска/минтай (уже есть) — **увеличить число рецептов**, не дублируя одну и ту же связку.
- **Бобовые:** чечевица, фасоль (в словаре ключей при необходимости расширять отдельной задачей — иначе primary может «не видеть» бобовые).
- **Овощи:** брокколи, цветная капуста, кабачок — уже встречаются; нужно больше **не-суповых** ужинов с ними в первых строках ингредиентов.
- **Мясо:** говядина (мало), индейка (много) — баланс.

### 4.4 Типы блюд

- **Finger food** (+8…+12) — по слотам snack / dinner.
- **Запеканки** без яйца или с заменой — под аллергию.
- **Рагу / тушёное** (не только суп) — +10…+15 на dinner/lunch replacement logic в будущем осторожно (обед остаётся супом).
- **Супы** с разной базой (бобовые, рыба, грибной только если возрастные правила позволяют) — см. инварианты продукта.

## 5. Риски при наполнении

- **Возраст:** не смешивать жёстко прикорм &lt;12 с пулом 12–36; проверять `min_age_months` / `max_age_months`.
- **Обед = суп:** рецепты без `is_soup` и без суповой лексики в названии **не попадут в lunch** на Edge.
- **Аллергии / dislikes:** новые рецепты с «спорными» ингредиентами (орехи, мёд, мед) — помечать и тестировать токенами.
- **Дубли по названию:** попадут под `exclude_title_keys` / variety — лучше **вариативные названия**.
